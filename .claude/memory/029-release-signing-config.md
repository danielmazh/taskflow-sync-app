Insight: Release signing wired through `android/app/build.gradle.kts` reading `rootProject.file("key.properties")` (= `android/key.properties`). Required imports at the top of the Kotlin DSL file: `import java.util.Properties` + `import java.io.FileInputStream` — Groovy auto-imports them, Kotlin DSL does NOT. A `signingConfigs.create("release") {…}` block is always evaluated; the buildTypes.release.signingConfig switches between `signingConfigs["release"]` (when key.properties exists) and `signingConfigs["debug"]` (fallback for `flutter run --release` without the keystore). `android/.gitignore` (Flutter template) already covers `key.properties`, `*.jks`, `*.keystore`; the top-level `.gitignore` also lists `/android/key.properties`, `*.jks`, `*.keystore` as defense-in-depth. Operator-owned upload keystore lives at `~/taskflow-upload-keystore.jks` (alias `upload`, 2048-bit RSA, 10,000-day validity). The keystore + passwords NEVER enter the repo. `pubspec.yaml` currently at `version: 1.0.0+1`; the `+N` versionCode must monotonically increase on every Play upload (Play rejects ties or decreases). Build-side flows documented in `docs/RELEASE-SIGNING.md`; release-build command is `flutter build appbundle --release`. Release SHA-1 for the Android OAuth client comes from Play Console → Setup → App signing AFTER the first AAB upload — until then only the debug SHA-1 (memory 020) is registered.
Rule: NEVER commit `*.jks`, `*.keystore`, or `key.properties`. If anyone reorders or removes the `import java.util.Properties` / `java.io.FileInputStream` lines at the top of `android/app/build.gradle.kts`, the release-signing block stops resolving — keep them. Always bump pubspec `+N` before a Play upload, even for an identical user-facing version.
