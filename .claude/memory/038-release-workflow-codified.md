Insight: rules.md now has a "Release management & environments" section. Branch roles: `master` = Play production mirror, never a dev target; only updated by an operator-approved `pre-master → master` merge at a real store release, tagged `prod-vX.Y.Z+N`, immediately followed by AAB rebuild + upload. `pre-master` = integration trunk; feature branches merge back `--no-ff` ONLY after the operator's on-device (S25 Ultra) smoke test passes — analyze + test green is necessary but not sufficient. Folded chronologically; no history-tidying reorders. Any build feeding a `master` push, and any push that advances `master`, needs explicit at-the-moment operator approval — standing approvals do not carry. No force-push on either trunk. Claude/local releases live on `pre-master` only (bump pubspec, signed APK to gitignored `.claude/releases/`, tag `claude-vX.Y.Z+N`) — they never enter Play. `versionCode` is one monotonic integer across production + Claude; reuse or lowering forbidden. Persisted state is JSON in `shared_preferences` only — there is **no database** by design (introducing one is Tier-B). Changes must be additive with null-defaulting `fromJson` (the `Task.label` pattern); key renames/removals need a migration shim or a Tier-B halt.
Rule: Treat `master` as read-only mirror of Play. Never branch features off it, never build Claude artifacts from it, never push to it without at-the-moment operator approval. Every `versionCode` is consumed once any build ships it. Persisted JSON changes stay additive with null defaults; no DB without a Tier-B decision.
