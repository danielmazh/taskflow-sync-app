Phase 4 — Google Sign-In + one-way Google Calendar export. Approved dependency cluster (the ONLY new deps): google_sign_in 7.2.0, googleapis 16.0.0, extension_google_sign_in_as_googleapis_auth 3.0.0 (transitive: googleapis_auth 2.3.2, google_sign_in_android 7.2.13). Anything outside this cluster = Tier-B halt.
Web client ID is a public identifier and lives as a const in lib/main.dart (_googleWebClientId). The web client SECRET is in .claude/env/ ONLY, never in source/APK/git. Android client is auto-matched by package+SHA-1 (debug SHA-1 in mem 018).
Scope: only `https://www.googleapis.com/auth/calendar.events` (least privilege).
Model change (authorized exception, surgical): Task gains `calendarEventId: String?`. fromJson defaults missing → null so all pre-Phase-4 saved tasks load. No other model/persistence/notification changes.
CalendarSyncService is decoupled — wired into TaskStore via onSyncUpsert / onSyncDelete callbacks, NEVER referenced directly by the store. TaskStore._syncReconcile mirrors _reconcile for notifications: dated+undone → upsert (returns event id, rounded back into task + persisted); else delete (clears the id). resyncAll() is called once on a fresh authorization (back-fill).
Best-effort contract: every CalendarSyncService method swallows exceptions and returns null/no-op; the store's _syncReconcile wraps in try/catch too. Network/auth/quota failures cannot break a store mutation.
Sign-in is OPTIONAL: app remains fully usable signed-out. attemptLightweightAuthentication() is gated by a `calendar_connected` shared_preferences flag set on successful connect(), cleared on disconnect() — so a fresh install NEVER auto-prompts. UI: a calendar IconButton in the Home AppBar shows a PopupMenu — when disconnected it offers "Connect Google Calendar"; when authorized it shows the email + a "Disconnect" entry. A small accent dot overlays the icon when authorized.
Sign-out leaves existing calendar events as-is (does NOT delete the user's events).
Sync rules in TaskStore: add → if dated+undone, upsert; toggle to done OR delete OR clear due → delete the stored eventId; toggle back to undone with due → upsert (creates a new event). On 404/410 on patch, service falls back to insert (self-heals when user deleted the event in Google Calendar).
Event details (Tier-A): summary = task.title; description = task.note; start = effectiveDueAt (UTC); end = start + 30 min (no per-task duration in our model).
OAuth-console GATE — known: with the consent screen in "Testing" mode the Web OAuth client only accepts users listed in Test Users on the consent screen. First connect attempt on a non-listed account returns access_denied:403; service handles it gracefully (logs, leaves disconnected). To pass DoD on a real device the operator must add the Gmail to Test Users in GCP → APIs & Services → OAuth consent screen → Test users. Refresh tokens then last ~7 days while still unverified — periodic re-auth is expected, not a bug.
Tests: pure store/model tests with a MockSync verify the upsert/delete decision logic, return-id round-trip, fail-path (upsert returns null → eventId untouched), and back-fill via resyncAll. 69 tests total, all green; no real network in test.
