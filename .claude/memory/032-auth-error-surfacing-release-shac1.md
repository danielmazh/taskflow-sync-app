Insight: Sign-in failed silently on the v1.0.0+1 release APK (picker opens, account picked, nothing visible). Root cause is **GCP-side**: the Android OAuth client only has the **debug** SHA-1 (`B0:10:0F:60:A7:1A:DD:C3:8E:F7:B2:61:1E:1F:53:60:13:85:87:65`, memory 020) registered; the upload-keystore SHA-1 (`6E:77:C3:89:6A:CD:ED:A7:AB:C9:2E:CF:A8:B7:8E:66:60:04:A3:76`) is not. google_sign_in v7 throws `GoogleSignInException(clientConfigurationError)` after Play Services rejects the OAuth call; pre-4d `CalendarSyncService.connect()` swallowed every exception via `debugPrint` and returned `void`, so the UI never knew anything went wrong. Phase 4d (this slot) makes `connect()` return a `ConnectResult { outcome, detail? }` with a `ConnectOutcome` enum (signedInAndAuthorized / signedInScopeDenied / canceled / signInClientConfigError / signInFailed / authorizationFailed / platformUnsupported). The AppBar account button's `_runConnect` maps each outcome to a SnackBar; failures expose a "Details" action that opens an AlertDialog with the GoogleSignInException code+description (selectable text) AND, for clientConfigurationError, the operator fix verbatim: register the release SHA-1 against the Android OAuth client for package `com.taskflowsync.taskflow_sync` in GCP → APIs & Services → Credentials, wait ~5min for propagation, retry. pubspec bumped to 1.0.0+2; rebuilt signed APK at `.claude/releases/taskflow-sync-v1.0.0+2.apk` (55.4 MB, apksigner-verified SHA-1 6E77…A376, signed by alias `upload`). The replaced Phase 6 APK was deleted. Tests 82 still green; analyze clean.
Rule: NEVER swallow auth failures silently. Any change to `CalendarSyncService.connect()` MUST preserve the typed `ConnectResult` return + the snackbar+dialog surfacing in `_AccountButton._runConnect`. The release-key SHA-1 registration is an admin-only GCP step (memory 022 / 026); the bot can ship a new APK but cannot make sign-in succeed on its own — always quote the operator fix in the report.
