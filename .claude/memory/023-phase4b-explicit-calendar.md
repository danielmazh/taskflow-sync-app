Insight: Phase 4b replaces Phase 4's automatic Calendar sync with **explicit, per-task export**. TaskStore.add/update/toggle no longer call upsert; `resyncAll()` and the main.dart "authorized→true fires back-fill" listener are removed. Two new user-triggered store methods: `exportTaskToCalendar(id)` (upsert → round id into task → persist+notify) and `removeTaskFromCalendar(id)` (delete → clear id → persist+notify). `delete()` still removes the calendar event (no orphans) — this is the only remaining automatic calendar call. AddTaskSheetResult gains `addToCalendar: bool` (only true on the add flow when connected+authorized+due-time-set, default off). Home screen renders a per-row trailing: small `event_available_rounded` indicator when `calendarEventId != null`, plus a `PopupMenuButton` whose items depend on state — `Export to calendar` for un-exported dated tasks, `Update in calendar` + `Remove from calendar` for exported tasks. All actions gated on `CalendarSyncService.connection.authorized` via `ValueListenableBuilder` so the menu disappears live when the user disconnects. Snackbar feedback on every action; errors are best-effort and never throw out. Test count 73 → all green; analyze clean.
Rule: Editing or completing a task MUST NEVER touch Google Calendar. Only `delete()` + the two explicit methods may. If you re-introduce an automatic upsert anywhere in TaskStore, you are violating the Phase 4b operator decision — halt and ask.
